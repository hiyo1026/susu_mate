# AGENTS.md

## 1. このファイルの役割

このリポジトリで作業するCodex・開発者は、変更前に必ず次を確認すること。

1. `AGENTS.md`
2. `README.md`
3. 既存の `index.html`
4. 既存の `style.css`
5. 既存の `script.js`

このプロジェクトは、学生プロジェクトの走り始めを支援する学習メディア  
**「すすメイト / susumate」** のWebサイトである。

Codexは、既存コードを破棄して新しいフレームワークで作り直すのではなく、  
現在のHTML・CSS・JavaScriptを土台として段階的に複数ページ化すること。

---

## 2. 最優先するユーザー体験

このサイトが目指す状態変化は次の通り。

- 利用前：
  - プロジェクトをやることになったが、何から手をつければよいか分からない
  - 会議で何を話せばよいか分からない
  - 役割分担や計画の立て方が分からない
- 利用後：
  - プロジェクト全体の流れが見える
  - 次にすることが分かる
  - 学生だけでも進められそうだと思える

派手な演出や機能数よりも、  
**「今どこにいて、次に何をすればよいか分かること」**を優先する。

## 3. 現在の技術構成

このプロジェクトでは、当面次の構成を維持する。

- HTML
- CSS
- Vanilla JavaScript
- 複数の静的HTMLページ
- GitHub Pagesなどで公開可能な静的サイト

### 重要

明示的な依頼がない限り、次を行わないこと。

- Astro、React、Vue、Next.jsなどへの全面移行
- npm環境やビルド工程の新規導入
- 既存トップページの全面作り直し
- 既存のデザインテーマの大幅変更
- 既存CSSクラスの理由なき全置換

フレームワーク移行が必要だと判断した場合も、勝手に実行せず、  
まず理由・メリット・影響範囲を報告すること。

## 4. 既存ファイルの扱い

### `index.html`

現在のトップページであり、サイト全体のデザインと文章表現の基準とする。

既存の主な構成：

- ヘッダー
- ヒーロー
- サイト紹介
- 学生プロジェクトの悩み
- ロードマップ
- 記事紹介
- CTA
- フッター

既存の内容を活かしながら、サイトマップに合わせてリンク先と表示内容を更新する。

### `style.css`

サイト全体の共通スタイルとして使う。

既存の主な特徴：

- 夜から夜明けへ進む空・宇宙の世界観
- ダークネイビーを基調とした配色
- オーロラ色と夜明け色のアクセント
- レスポンシブ対応
- モバイルナビゲーション
- `prefers-reduced-motion` 対応
- カード、ロードマップ、ボタンなどの共通スタイル

新しいページでも、原則として同じCSSを読み込む。  
ページ固有のCSSが増える場合も、まず `style.css` 内でセクションを分けて管理する。

CSSが過度に長くなった段階でのみ、以下のような分割を検討する。

```text
styles/
├── common.css
├── top.css
├── tools.css
├── basics.css
├── articles.css
└── about.css
```

分割する場合は、すべてのHTMLの読み込み先を同時に更新すること。

### `script.js`

サイト全体で共通して使うJavaScriptとして扱う。

既存の主な機能：

- フッターの年表示
- スクロール後のヘッダー表示変更
- モバイルナビゲーション
- Escapeキーによるメニュー終了
- スクロール表示アニメーション
- ロードマップの進行演出
- `prefers-reduced-motion` 対応

新しいページに存在しない要素を取得してもエラーにならないよう、  
現在と同様に要素の存在確認を行うこと。

ページ固有機能を追加する場合は、ページを判別してから初期化する。

```js
const page = document.body.dataset.page;

if (page === 'icebreak-gacha') {
  // アイスブレイクガチャ固有処理
}
```

### パスのルール

ルート直下のHTML：

```html
<link rel="stylesheet" href="style.css">
<script src="script.js"></script>
```

1階層下のHTML：

```html
<link rel="stylesheet" href="../style.css">
<script src="../script.js"></script>
```

リンク例：

ルート直下から個別ページへ：

```html
<a href="tools/kickoff-roadmap.html">キックオフロードマップ</a>
```

個別ページからトップへ：

```html
<a href="../index.html">ホーム</a>
```

個別ページから別カテゴリ一覧へ：

```html
<a href="../articles.html">記事</a>
```

公開先のリポジトリ名をHTML内へ直接書かないこと。

悪い例：

```html
<a href="/susumate/tools.html">
```

良い例：

```html
<a href="tools.html">
```

または、階層に応じた相対パスを使用する。

## 9. デザイン方針

### テーマ

**旅路 × 空・宇宙**

既存デザインの「夜の空から夜明けへ進む旅」の方向性を維持する。

- 星：小さな気づき、目印
- 軌道：プロジェクトの進行
- 通過点：決めること、行動すること
- 夜明け：迷いが減り、見通しが持てる状態
- 地図：進め方を共有する道具

### 現在の配色

`style.css` のCSS変数を基準にする。

主な色：

- ダークネイビー
- 深い青
- 夜明けのオレンジ
- オーロラのミント
- 星の白
- 青紫系の本文色

旧資料にある別のカラーパレットを勝手に混在させない。  
色変更を行う場合は、まずCSS変数を変更する。

### 文章のトーン

- 編集部が伴走するような語り口
- 親しみやすい
- 幼すぎない
- 社会人向けサービスのように堅すぎない
- 不安を煽らない
- 命令口調にしない
- 「まずはここから」「5分で確認」など行動のハードルを下げる

## 10. HTMLルール

- すべてのページで `lang="ja"` を指定する
- ページごとに固有の `<title>` を付ける
- ページごとに固有の `meta description` を付ける
- 見出し順を守る
- `<main>` は1ページに1つ
- ナビゲーションには `aria-label` を付ける
- 現在ページには `aria-current="page"` を付ける
- 装飾SVGには `aria-hidden="true"` を付ける
- フォームには必ず `<label>` を付ける
- ボタンは動作、リンクはページ遷移に使う
- クリックできない要素をボタンのように見せない
- 「こちら」だけのリンク文言を避ける
- 画像には適切な `alt` を付ける
- 装飾画像の `alt` は空にする
- 全ページにスキップリンクを置く
- Header、main、footerの基本構造を揃える

各ページの `<body>` にはページ識別用属性を付ける。

```html
<body data-page="tools">
```

個別ツール例：

```html
<body data-page="icebreak-gacha">
```

## 11. CSSルール

- 既存CSS変数を優先して使う
- 新しい色を直接多数追加しない
- ページ固有クラスにはカテゴリ名を付ける

例：

```css
.tools-index {}
.tool-detail {}
.basics-index {}
.article-detail {}
.about-page {}
```

- `!important` を原則使わない
- 固定幅だけに依存しない
- `rem`、`clamp()`、Grid、Flexboxを使う
- 375px、768px、1280pxで確認する
- 横スクロールを発生させない
- フォーカス表示を消さない
- 色だけで状態を伝えない
- `prefers-reduced-motion` を維持する
- 既存の `.container`、`.section`、`.btn`、`.card-grid` などを再利用する
- 同じ見た目の新しいクラスを重複して作らない

---

## 12. JavaScriptルール

- Vanilla JavaScriptを使用する
- 外部ライブラリは必要になるまで追加しない
- ページに存在しない要素を前提にしない
- 取得した要素の存在確認を行う
- グローバル変数を増やさない
- 即時関数またはモジュール内に処理を閉じ込める
- エラーが起きても他の共通機能を止めない
- キーボード操作に対応する
- 動きを減らす設定を尊重する
- localStorageを使う場合は、保存目的と削除方法を明確にする
- 入力内容を外部へ送信しない
- ツールの結果はHTMLへ安全に反映し、ユーザー入力を `innerHTML` へ直接渡さない

悪い例：

```js
result.innerHTML = userInput;
```

良い例：

```js
result.textContent = userInput;
```

## 14. 作業手順

Codexは、各タスクで次の順に進める。

1. `AGENTS.md` と `README.md` を読む
2. `index.html`、`style.css`、`script.js` を確認する
3. 既存デザインと機能を把握する
4. 変更対象と影響するリンクを洗い出す
5. 必要最小限の変更を行う
6. 全ページのリンクを確認する
7. 375px、768px、1280pxで表示を確認する
8. JavaScriptエラーを確認する
9. 変更点と確認結果を日本語で報告する

### 変更前に確認すること

- 既存のCSSクラスを再利用できないか
- 既存のJavaScript機能へ影響しないか
- 下層ページの相対パスが正しいか
- ヘッダーとフッターの更新漏れがないか
- モバイルナビゲーションが動くか

## 15. 確認方法

ビルド工程は不要。

簡易サーバーを使って確認する。

Pythonが利用できる場合：

```bash
python3 -m http.server 8000
```

ブラウザで開く：

```text
http://localhost:8000/
```

確認項目：

- トップページが表示される
- CSSが読み込まれる
- JavaScriptエラーがない
- モバイルメニューが開閉する
- すべての主要ページへ移動できる
- 下層ページからトップへ戻れる
- 画像パスが壊れていない
- 現在ページ表示が正しい
- 登録・ログイン要素がない
- 横スクロールがない
- キーボードで操作できる
- 動きを減らす設定でも利用できる

リンク確認には、必要に応じて次を使用する。

```bash
grep -R 'href="' . --include="*.html"
grep -R 'src="' . --include="*.html"
```

2. 既存コードと既存デザインを壊さないこと
3. 学生が迷わず使えること
4. サイトマップどおりに移動できること
5. アクセシビリティ
6. スマートフォンでの使いやすさ
7. 見た目の華やかさ
8. 技術的な新しさ
